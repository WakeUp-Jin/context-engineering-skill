# 协同 Agent 与自主 Agent 交互形态

## 两种形态

### 协同 Agent

人与 Agent 在同一个"场域"中协作完成任务。人能观察 Agent 的行为，Agent 也能观察人的行为，形成双向信息流。

典型平台：Cursor、Windsurf、YouMind、Lovart。

**核心特征**：
- 人类可以实时观察 AI 的工作过程
- 发现偏差可以及早纠正
- 关键步骤需要人类批准（如执行终端命令、写入文件）
- 输出采用"商量式"而非"命令式"

### 自主 Agent

Agent 主导完成任务，人仅提供任务输入。对模型能力要求更高。

**核心特征**：
- 人只提供初始需求
- Agent 独立完成所有步骤
- 对上下文精确性要求极高（没有人纠错）
- 容忍度更低（错误无法及时发现）

## 场域理论

场域是人和 AI 之间的有效交流空间。好的场域实现双向信息流：

```
人 → Agent 的信息流：
├── 用户输入（显式意图）
├── 预定义规则（如 .cursorrules）
├── 状态上下文（代码片段、错误信息、日志）
└── 行为观察（用户在编辑器中的操作）

Agent → 人 的信息流：
├── 输出文本和代码
├── 审核请求（需要确认的操作）
├── 状态展示（正在执行什么工具）
└── 执行进度和错误信息
```

## 两类上下文

### 意图上下文（规定性）

描述期望结果——"我想要什么"：

```
- 用户的自然语言描述
- 系统提示词中的规则和约束
- 结构化输出的格式要求
```

### 状态上下文（描述性）

描述当前状态——"现在是什么情况"：

```
- 当前代码的内容和结构
- 编译/运行时错误
- 已执行操作的结果
- 文件系统状态
```

协同 Agent 可以同时利用两类上下文；自主 Agent 主要依赖意图上下文，状态上下文需要自己收集。

## Cursor 的协同机制

```
人 → 模型：
├── 用户输入
├── @-引用的文件和符号
├── .cursorrules 预定义规则
├── 当前打开文件的内容
├── 光标位置和选中区域
├── 终端输出
└── lint 错误

模型 → 人：
├── 代码差异（diff）展示
├── 文件修改的确认/拒绝机制
├── 工具调用的实时展示
└── 思考过程的透明展示
```

## 自主 Agent 的局限

1. **缺乏精确上下文**：无法实时获取用户的意图变化
2. **个性化差异大**：不同用户对"好"的定义不同
3. **容忍度低**：没有人工纠错，小错误可能累积
4. **反馈不足**：无法从用户的实时反应中学习

## 协同 Agent 开发方向

1. **建立协作平台**：让人和 Agent 共处同一个工作空间
2. **收集完整上下文**：用户行为、编辑历史、环境状态
3. **工具的完整说明**：每个工具都有清晰的描述和使用指南
4. **精准的上下文筛选**：不是所有信息都需要注入，过滤无关信息
5. **审核机制**：重要操作需人确认，建立信任
6. **渐进信任**：随着协作深入，逐步放开审核（如 ApprovalMode 从 DEFAULT 到 AUTO_EDIT）

## 设计建议

**当前阶段优先协同 Agent**：

模型能力还不足以完全自主工作。协同模式下，人的参与可以：
- 补充模型缺失的上下文
- 及时纠正方向偏差
- 确认关键决策
- 提供主观判断（审美、优先级等）

**面向未来准备自主 Agent**：

在协同 Agent 的基础上，逐步减少人的介入：
- 完善上下文自动收集能力
- 增强错误自检和恢复机制
- 建立更好的评估体系（参见 [evaluation/](../evaluation/) 模块）
- 提高工具输出的可靠性
